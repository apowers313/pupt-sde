# Proposed SDE Prompts for pupt-lib

## Summary

11 structured prompts covering the software development lifecycle, selected based on
analysis of 9,224 real developer prompts across 148 repositories and 25 independent
prompt collections.

## Selection Criteria

A prompt earns its place in pupt-lib only when a pre-built structured template
**meaningfully outperforms someone just asking the AI to do the thing ad-hoc**. We
applied four tests:

1. **Completeness matters** — there's a checklist of things to systematically verify,
   and humans skip items without structure
2. **Methodology isn't obvious** — there's a better process than what someone would
   naturally do
3. **Output format matters** — the result needs a specific structure to be useful
4. **The task has phases** — steps that should happen in order, where skipping ahead
   produces worse results

Tasks where the AI already performs well with minimal instruction (explaining code,
writing simple functions, generating commit messages) were excluded regardless of
popularity.

## The 11 Prompts

### 1. Requirements Clarification

**SDLC Phase:** Requirements

**Input:** Vague feature request, bug report, or stakeholder ask

**Output:** Clarified requirements, acceptance criteria, identified ambiguities,
out-of-scope items, dependencies, open questions

**Why structure adds value:** Developers routinely start coding without fully
understanding the problem. A structured prompt forces identification of ambiguities
and edge cases before any design or code is written. This is the highest-leverage
prompt in the set — catching a requirements misunderstanding here saves days of
rework downstream.

**Collection evidence:** No direct "requirements clarification" category exists in the
collections. The closest analog is plan/spec/PRD (15/25 repos, 59
files), but those focus on implementation planning rather than requirements
elicitation. The absence itself is notable — it suggests developers jump from feature
request to implementation without a structured requirements step.

**Why it wasn't in early drafts:** Our initial analysis focused on prompt collection
popularity, where "requirements" is underrepresented. The gap was identified when we
mapped the proposed prompts against SDLC phases and found nothing covering the front
of the cycle.

### 2. Design / Architecture

**SDLC Phase:** Design

**Input:** Clarified requirements (output of prompt #1)

**Output:** Architecture decisions with trade-offs, component boundaries, data model,
API surface design, technology choices with rationale

**Why structure adds value:** Architecture decisions (queue vs. polling, SQL vs. NoSQL,
monolith vs. service, REST vs. GraphQL) happen before implementation planning and
have outsized impact on project success. A structured prompt that forces explicit
consideration of trade-offs, scalability implications, and failure modes produces
better decisions than ad-hoc "how should I architect this?"

edge cases to handle, test strategy, rollback considerations

structure, developers dive into code and discover missing pieces halfway through —
an undocumented dependency, an unconsidered edge case, a missing migration. The
structured prompt front-loads this discovery.

### 4. Test Generation

**SDLC Phase:** Testing

**Input:** Code under test

**Output:** Systematic tests covering happy path, edge cases, boundary conditions,
error cases, null/empty inputs, concurrency scenarios, state transitions

**Why structure adds value:** When someone says "write tests," they get happy-path
coverage. A structured prompt that explicitly requires boundary conditions, error
cases, and interaction testing produces dramatically more thorough test suites. The
AI is capable of identifying edge cases humans miss — it just doesn't do it without
being specifically asked.

### 9. Debug / Root Cause Analysis

**SDLC Phase:** Maintenance / Incident Response

**Input:** Error report, stack trace, failing test, or production incident description
plus system context (architecture, available logging, recent changes)

**Output:** Structured diagnostic: reproduction steps, isolation of the fault
boundary, ranked hypotheses with evidence, verification plan for each hypothesis,
proposed fix, regression test to prevent recurrence

**Why structure adds value:** Debugging is the most universal developer activity (23/25
collections), yet developers under pressure default to guess-and-check rather than
systematic diagnosis. A structured prompt enforces the
reproduce → isolate → hypothesize → verify → fix → regression-test methodology that
experienced debuggers use but most developers skip. The "isolate" and "hypothesize"
steps are where structure matters most — they prevent developers from jumping to
fixes before understanding the problem.

**Design consideration — requires system context:** Unlike other prompts in this set,
Debug/Root Cause benefits significantly from project-specific context: what
monitoring exists, what the architecture looks like, what changed recently, what
logging is available. The prompt should accept these as optional parameters and
degrade gracefully when they're not provided. Without system context, it still
provides the structured methodology; with it, it can give targeted diagnostic steps.

### 11. PR Description

**SDLC Phase:** Release

**Input:** Diff, commit history, related issues

**Output:** Structured PR description with: summary of why (not what), breaking
changes, migration steps if needed, test plan, reviewer guidance (what to focus on)

**Why structure adds value:** High frequency (multiple times per week for active
developers) and the structured output format ("why not what," breaking change
callouts, test plan, reviewer focus areas) is something developers consistently
forget without a template. Small per-use improvement multiplied by high frequency
equals significant cumulative value.

**Design decision — commit messages excluded:** Commit message generation is
commoditized by existing tools (built into Claude Code, GitHub, every IDE plugin).
The marginal value of a pupt-lib prompt is near zero.

## Research Basis

This proposal is grounded in analysis of real developer prompts, not theoretical
SDLC models.

### Source Data

- **9,224 prompt files** across **148 repositories** in **12 categories**
  (slash-commands, skills, copilot-prompts, system-prompts, cicd-prompts,
  fabric-patterns, framework-prompts, hooks, ralph-wiggum, spec-driven,
  se-activity, configuration)
- Cataloged in `design/sample-prompts/INDEX.md` (generated by
  `design/sample-prompts/build-index.py`)
- Raw metadata in `design/sample-prompts/index-data.json`

### Analyses Performed

1. **Command/function analysis** (`design/sample-prompts/analyze-commands.py`):
   Compared filenames across the 25 largest independent collections to find
   universal functions. Key finding: Debug/fix (23/25 repos), Test/TDD (22/25),
   Explain/understand (20/25), Refactor (20/25), Security audit (19/25) were the
   most universal. Code generation appeared in only 6/25 repos.

2. **Content analysis** (`design/sample-prompts/analyze-content.py`): Deep regex
   analysis of prompt content across all 9,224 files. Key findings:
   - Code maintenance activities dominate over code generation (debug 33%,
     documentation 35%, explain 21% vs. code generation 12.4%)
   - "Verify/validate" is the #1 constraint (37% of files)
   - Only 14% of prompts assign a persona
   - The most common persona is "helpful assistant" (6%)

3. **Structural analysis** (`design/sample-prompts/build-index.py`): File format
   distribution (59.8% markdown), complexity distribution, technology keyword
   frequency, SE activity coverage by category.

### Key Insight

The most popular prompt categories (debug, explain, code generation) are NOT
necessarily the highest-value structured prompts. Popularity measures what
developers do frequently; value measures where structure beats ad-hoc. These are
different axes. Debug is the most universal need but benefits most from structure
when paired with system-specific context. Code explanation is extremely common but
the AI already handles it well unstructured.

## Design Principles for Implementation

1. **Composability:** These 11 prompts should chain together with consistent
   input/output contracts. Natural workflows:
   - **Feature flow:** Requirements → Design → Plan → [Code] → Test → Review → PR
   - **Bug flow:** Debug/Root Cause → [Fix] → Test → Review → PR
   - **Maintenance flow:** Performance/Security Analysis → Refactor → Test → Review
   - **Documentation flow:** Code → Documentation audit → Generate missing docs

2. **Parameterization over duplication:** Prompts should accept parameters for
   language, framework, project conventions, and strictness level rather than
   creating technology-specific variants.

3. **Checklist completeness over conversational flexibility:** The value of these
   prompts is in forcing systematic coverage. They should feel more like structured
   audits than open-ended conversations.

4. **Verification built in:** Where possible, prompts should include a verification
   step (refactor verifies behavioral equivalence, test generation verifies edge
   case coverage, security audit verifies all OWASP categories addressed).

5. **Context-aware degradation:** Prompts like Debug/Root Cause that benefit from
   system-specific context should accept it as optional parameters and still provide
   value (structured methodology) when context is unavailable.
